
# Overall Project Architecture

I would split the project into three major modules instead of two.

```
                ┌───────────────────────────┐
                │ Website Knowledge Engine  │
                │  (Crawler)                │
                └────────────┬──────────────┘
                             │
                 Creates Website Knowledge
                             │
         graph.json
         flows.json
         sqlite.db
         embeddings
         screenshots
                             │
                             ▼
                ┌───────────────────────────┐
                │ Knowledge Index           │
                │ Retrieval Layer           │
                └────────────┬──────────────┘
                             │
                      User Prompt
                             │
                             ▼
                ┌───────────────────────────┐
                │ LLM Browser Agent         │
                │ (Playwright)              │
                └───────────────────────────┘
```

The crawler never automates user requests.

It only learns the website.

The LLM only performs tasks.

---

# Why Playwright?

I recommend **Playwright**.

Reasons:

✓ Chromium

✓ Chrome

✓ Edge

✓ Firefox

✓ Works locally

✓ Stable

✓ Supports downloads

✓ Handles iframes

✓ Supports Shadow DOM

✓ Network interception

✓ Storage state

✓ Multiple tabs

✓ File upload

✓ Good Python support

Nearly every modern browser agent eventually uses Playwright underneath.

# Stage 7 — Duplicate Detection

One of the hardest problems.

Example

```
Playlist 1

Playlist 2

Playlist 3

Playlist 4
```

These should NOT become

```
4 workflows
```

Instead

```
Playlist Detail Page

Parameter

playlist_id
```

The crawler needs template detection.

It compares

* DOM

* Layout

* XPath

* URL

* Visual similarity

* LLM similarity

Then clusters pages.

Instead of

```
/playlist/123

/playlist/456

/playlist/789
```

store

```
/playlist/{id}
```

Exactly like routing frameworks.

This saves thousands of nodes.

# Stage 13 — Weekly Updates

Rather than recrawling everything, implement an incremental crawl:

1. Load the previous graph and known templates.
2. Start from the homepage and authenticated dashboard.
3. Detect changed pages using DOM hashes and visual hashes.
4. Re-crawl only changed branches and newly discovered paths.
5. Update embeddings only for modified content.
6. Merge changes into the existing graph while preserving stable node IDs.

This will reduce crawl time dramatically once the initial knowledge base has been built.

# One Additional Capability I'd Add

The feature I believe will make this system significantly more reliable is **Action Contracts**.

Instead of recording only:

```
Click Settings

↓

Click Profile

↓

Click Change Password
```

store a contract for every action:

```json
{
  "action": "Open Change Password",
  "preconditions": [
    "User is logged in",
    "Profile menu is visible"
  ],
  "target_state": "Change Password Form",
  "success_indicators": [
    "Old Password field visible",
    "New Password field visible",
    "Save button enabled"
  ],
  "failure_indicators": [
    "Access Denied",
    "Session Expired"
  ],
  "recovery_actions": [
    "Re-login",
    "Navigate back to Dashboard"
  ]
}
```

This is extremely valuable because websites evolve. If a menu moves or a layout changes, the LLM doesn't blindly replay clicks; it verifies whether the expected state has been reached and can recover when it hasn't. That makes the agent far more robust than a simple recorded macro.

## My recommendation for implementation

I would build this in four phases:

1. **Core crawler:** Browser control, DOM extraction, graph building, duplicate detection, and storage.
2. **Semantic layer:** Workflow extraction, form understanding, embeddings, and LLM-guided exploration.
3. **Execution layer:** The LangGraph-based Playwright agent that retrieves workflows and performs user-requested tasks with state validation.
4. **Advanced capabilities:** Incremental crawling, vision fallback, network analysis, API discovery, distributed crawling, and self-healing when the UI changes.

I believe this architecture can scale beyond FOSCOS to many government portals and enterprise web applications while keeping the execution agent simple: it reasons over a structured knowledge base instead of rediscovering each website from scratch every time.
